# LXNS 收藏品获取条件共性分析

> 数据时间：2026-07-19 00:47（Asia/Shanghai）
>
> 数据范围：LXNS 舞萌公共 API 的 `trophy`、`icon`、`plate`、`frame` 四类列表，均带 `required=true`。原始响应位于 `data/lxns-collections/`。

## 结论

获取条件存在明显共性，但这个共性只适用于带结构化 `required` 的 1,730 件收藏品（36.26%），不能代表全部 4,771 件收藏品。

- 结构化部分本质上都是“曲目集合 + 难度集合 + 可选成绩门槛”的谱面成绩规则，可以用玩家成绩自动计算。
- 3,041 件（63.74%）没有 `required`。它们主要通过角色觉醒、区域/活动获得、赠送或其他账号状态取得，公共响应只给自然语言描述，无法仅靠成绩快照可靠计算。
- 因而收藏品进度不应使用单一规则引擎。至少应区分“成绩可计算”“持有状态可确认”“条件仅文本”“条件异常/未知”四种状态。

## 全量概况

| 类型 | 总数 | 带 `required` | 无 `required` | 结构化覆盖率 | 条件对象数 |
|---|---:|---:|---:|---:|---:|
| 称号 `trophy` | 2990 | 1557 | 1433 | 52.07% | 1559 |
| 头像 `icon` | 1041 | 18 | 1023 | 1.73% | 18 |
| 姓名框 `plate` | 397 | 105 | 292 | 26.45% | 110 |
| 背景 `frame` | 343 | 50 | 293 | 14.58% | 50 |
| **合计** | **4771** | **1730** | **3041** | **36.26%** | **1737** |

1,723 件结构化收藏品只有一个条件对象，另有 7 件包含两个条件对象。双条件主要用于把 BASIC～MASTER 与 Re:MASTER 拆开表达，并不是两套互斥的获得方式。

## 结构化条件的共同模型

对全部 1,737 个 `required` 对象逐字段检查后，得到以下不变量：

1. 每条都有 `songs`，且至少包含一首曲目；曲目均为带 `id`、`title`、`type` 的对象。
2. 每条都有 `difficulties` 数组；空数组表示描述未限定具体难度，不能当成“没有曲目条件”。
3. 除曲目与难度外，只会出现 `rate`、`fc`、`fs` 中至多一个成绩门槛。
4. 未发现其他结构化条件字段。

| 条件形态（按实际字段） | 条件对象数 | 占比 |
|---|---:|---:|
| `songs + difficulties + fc` | 1144 | 65.86% |
| `songs + difficulties` | 390 | 22.45% |
| `songs + difficulties + rate` | 169 | 9.73% |
| `songs + difficulties + fs` | 34 | 1.96% |

难度方面，1,013 条要求 BASIC～MASTER（`[0,1,2,3]`），468 条不限定难度（`[]`），其余是单一难度；没有其他难度数组组合。1,505 条条件只涉及一首歌，占 86.64%；其余条件从 2 首到 516 首，主要是版本牌子或全曲成就。

成绩门槛值域也很集中：

- `rate`：`a`、`s`、`ss`、`sss`。
- 连击类：`fc`、`fcp`、`ap`、`app`。
- 同步类：`fs`、`fsp`、`fsd`、`fsdp`。
- 无成绩门槛的 390 条，语义通常是游玩/通关指定曲目或曲组。

## 无结构化条件的主要共性

对 3,041 件无 `required` 收藏品的 `description` 做按优先级、互斥的文本归类，结果如下。该归类用于观察分布，不等同于官方枚举。

| 文本类别 | 数量 | 占无结构化条件比例 |
|---|---:|---:|
| 包含“觉醒” | 2145 | 70.54% |
| 区域/活动“ちほー…獲得” | 421 | 13.84% |
| 其他未归类描述 | 414 | 13.61% |
| 赠送 | 47 | 1.55% |
| 成绩语义但缺少 `required` | 10 | 0.33% |
| 其他“獲得” | 2 | 0.07% |
| 登录 | 2 | 0.07% |

这说明无结构化部分同样有共性：绝大多数不是谱面成绩条件，而是角色成长与活动地图状态。它们需要玩家收藏品持有列表、角色/地图进度或人工确认，无法从普通成绩记录反推。

## 数据质量与实现风险

- 同步条件字段不一致：83 条 `FULL SYNC` 系列条件把 `fs`、`fsp`、`fsd`、`fsdp` 值放在 `fc` 字段，另有 34 条使用独立 `fs` 字段。描述文本与值域可以确认其同步语义，消费端不能只按字段名判断。
- 10 件描述带 `ALL PERFECT` / `FULL SYNC` 等成绩语义，却没有 `required`；不能因为描述像成绩条件就自行猜出完整曲目、难度和门槛。
- 所有条目都有 `description`，但不代表描述都可执行。无结构化条件中有 69 件描述完全相同的“アットホームな職場です”，另有随机占位等文本。
- 不同类别的数值 ID 会重复；全局唯一键应使用 `(kind, id)`，不能只用 `id`。

## 对收藏品进度追踪的建议

1. 对 `required` 做统一归一化：保留曲目类型与难度；将落在 `fc` 中的同步枚举迁入统一的同步门槛。
2. 只有结构化条件完整时才显示自动计算进度；多条件对象按全部条件组共同完成处理。
3. 无 `required` 的收藏品只展示原始获取说明与“不可由成绩自动计算”，不要从描述生成伪精确进度。
4. 若能取得玩家完整持有列表，可直接确认“已拥有”，但“未拥有的完成百分比”仍需要相应角色、地图、活动等状态源。
5. 对上游缺失条件、字段错位和占位描述保留异常标记，避免错误结果覆盖最近一次可信快照。

## 验证方法

- 四份文件均已由 PowerShell `ConvertFrom-Json` 完整解析。
- 逐类核对顶层数组名称及数量，总数与分类求和一致。
- 遍历全部收藏品、全部 `required` 对象和全部曲目对象，统计字段集合、值域、条件对象数量、曲目数量及曲目类型。
- 计算并记录四份原始响应的 SHA-256，便于复查分析是否基于同一快照。
