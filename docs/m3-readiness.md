## 2. 范围冻结与实施顺序

### M3A：收藏、练习清单、本地标签与备份恢复

- 先定义稳定的收藏目标：歌曲使用规范化 `songId`，谱面使用 `songId + type + levelIndex`，不保存可从曲库重新生成的完整对象。
- 为用户数据建立独立 SQLite 表、schema version、迁移和事务测试，不复用可被清理的远端资源缓存。
- 备份使用带版本号的最小 JSON 契约和 Zod 校验；默认只包含用户主动创建的数据，不包含 token、cookie、账号密码、远端成绩快照或曲库缓存。
- 恢复流程必须先完整校验，再以事务执行；覆盖“合并/替换”、重复项、未知版本、损坏文件和跨端恢复。

实现状态：已完成。保持 5 个主 tab；个人数据为本机统一数据；清除时每次询问是否包含个人数据；恢复默认合并并允许显式替换；歌曲与谱面均可使用标签。

### M3B：B50、牌子和成绩结果导出

- 导出模型使用显式字段白名单，不直接序列化 provider 响应、Query cache 或 SQLite payload。
- 图片/文件中不得出现 token、cookie、隐藏字段或未获许可的素材；玩家标识默认最小化，并给用户预览确认。
- 编码前按 `apps/mobile/AGENTS.md` 核对 Expo SDK 54 文档，再验证文件生成、分享面板、取消分享和临时文件清理在 Expo Go 双端的真实行为。
- 当前未直接安装 `expo-sharing`、视图截图或媒体库依赖，不假定传递依赖可作为产品 API 使用。

### M3C：随机谱面、拟合定数与目标 Rating 推荐

- 先实现输入可固定、输出可复现的规则推荐；同一快照、偏好和随机种子必须产生相同结果。
- 每条推荐返回理由与参与排序的关键输入，明确区分“上分”“补短板”“随机探索”等模式。
- 拟合定数、谱面标签和弱项分析只有在来源、许可、更新频率与准确度可验证时启用；不满足门禁时保持关闭，不用推测数据补齐。
- 单元测试覆盖无成绩、数据稀疏、版本缺失、并列、过滤后为空和异常值。
