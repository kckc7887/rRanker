# 修复高分辨率与超长图导出失败

## 改动原因

成绩图片在预览中可以正常显示，但高分辨率、超长内容或两者同时出现时，导出会报 `drawViewHierarchyInRect was not successful`，严重时因原生位图瞬时内存过高导致应用退出。原实现把目标物理像素直接传给 iOS 截图 API，系统又按设备像素比放大，实际申请的位图远大于所选分辨率；2160px 输出仍容纳 50 行也进一步放大了内存峰值。

Expo Go 关于 Android 媒体库访问范围的警告属于宿主能力限制，与截图阶段的 `drawViewHierarchyInRect` 错误不是同一个问题。

## 具体实现

- 截图尺寸改为平台相关换算：iOS 使用 `目标物理像素 ÷ PixelRatio` 的逻辑尺寸，Android 保持原生实现要求的目标物理像素，最终 PNG 仍对应用户选择的 1080、1440 或 2160 宽度。
- iOS 的 1440/2160 输出或宽高比达到 1:4 的长图直接启用 `useRenderInContext`，避开不适合大视图的 `drawViewHierarchyInRect`；普通尺寸如仍返回该错误，也会自动切换渲染路径重试一次。
- 按 1080px、50 行的位图预算动态调整单页行数：1080px 最多 50 行、1440px 最多 28 行、2160px 最多 12 行。超出的内容继续自动分页，分区标题与排名规则不变。
- 预览分辨率说明同步显示当前分辨率的单页行数上限，预览分页与实际导出保持一致。
- 增加 iOS/Android 尺寸换算、大视图渲染策略和分辨率分页预算测试。

## 期望输出

- 高分辨率或超长成绩图不再创建超过目标尺寸数倍的原生位图。
- 2160px 长内容通过更多短页输出，避免导出报错或应用闪退。
- 导出图片保持所选目标分辨率，普通 1080px 图片和现有多页排名行为不回退。
- Expo Go 的媒体库警告不再被误判为截图渲染失败；完整相册权限仍在 development build 中验证。

## 实际输出

- 新增尺寸、渲染策略与分页测试全部通过；全量 Vitest 39 个文件、162 项测试通过。
- 全量 Jest UI 7 个文件、29 项测试通过；仅保留既有 VirtualizedList `act` 测试环境提示。
- TypeScript 检查通过；ESLint 0 错误，保留总览页既有 1 条 Hook 依赖警告。
- Android Expo 静态导出成功，Hermes bundle 正常生成。
- 真机高分辨率长图及相册写入仍需在 iOS development build 与 Android 真机上复测。
