# 修正 DXScore 数据链路与成绩卡布局

## 改动原因

- 成绩图片中的 DXScore 实际分与理论分发生错位：部分记录只显示理论分，且理论分被放进实际分槽位。
- `ScoreService` 使用普通曲库时拿不到详细物量，无法可靠计算理论 DXScore；两套成绩 schema 又允许 DXScore 字段直接缺失，掩盖了上游契约错误。
- DX/SD 标签参与歌曲头部纵向排版，三行歌名会把分隔线和整张卡片向下推；右下角状态标签字号偏小、透明底对比不足。

## 具体实现

- 将水鱼 `/player/records` 的 `dxScore` 与落雪 `/user/maimai/player/scores` 的 `dx_score` 统一定义为“玩家实际 DXScore”，要求字段明确存在并允许为 `null`；字段缺失由 schema 直接报错。
- `ScoreService` 的曲库依赖收紧为 `DetailedCatalogProvider`，统一请求带音符详情的曲库；理论分只按匹配谱面的 `notes.total × 3` 计算，并把详细曲库写入既有曲库缓存。
- 成绩查询版本从 3 升至 4，SQLite 快照结构版本从 4 升至 5，使缺少物量或实际分映射错误的旧缓存自动失效。
- HTML 将实际分、斜杠和理论分拆成独立节点，固定显示“实际 / 理论”；任一项缺失时只在对应槽位显示 `—`。
- DX/SD 标签绝对定位到歌曲信息右上角：SD 使用 `#3286E6` 蓝底白字，DX 使用白底、`#F2C36C` 金边和 `#FF8A00 → #FFD84A` 渐变字。
- 歌曲头部固定为封面高度，标题最多三行；移除标签参与纵向 flex 排版的行为，保证一行、两行、三行标题的分隔线位置一致。
- 状态标签保持“评价 → 寸 → FC → FS”顺序，提升字号与横向留白，改用高对比彩虹、金、绿、蓝、灰实底并固定在右下角。
- 补充水鱼与落雪实际分映射、字段缺失、详细物量 690 → 理论分 2070、`1836 / 2070` 组合、单项缺失、缓存迁移、DX/SD 样式、长标题等高和状态标签配色测试。

## 期望输出

- 水鱼与落雪账号均只从成绩 API 获取玩家实际分，并只从详细曲库物量计算理论分；两者不会互相替代、交换或省略。
- `notes.total = 690` 时理论分稳定为 2070，实际分为 1836 时图片显示 `1836 / 2070`。
- DX/SD 标签位于右上角且配色正确；三行或超长歌名不会改变卡片高度；底部状态标签清晰可读并保持固定顺序。
- 1080 与 2160 输出都能完整缩放进预览窗，五列布局和自动长图高度保持不变。
