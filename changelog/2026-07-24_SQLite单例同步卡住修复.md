# 缓解 SQLite 单例导致的同步卡住

## 改动原因

上一轮将 `rranker.db` 改为进程内单例后，安卓端好友码同步成绩明显变慢并提前超时。根因是单例连接上仍执行 `PRAGMA journal_mode = WAL`，且个人曲库使用 `withExclusiveTransactionAsync`（会另开连接）。两者并发时容易锁住原生 SQLite 队列，拖慢甚至卡住 score-hub 轮询。

## 具体实现

- 去掉 snapshot schema 初始化中的 `journal_mode = WAL`
- 增加 `runSerializedSchemaInit`，首启 snapshot / user-library 建表串行执行
- 个人曲库写操作改为同连接 `withTransactionAsync` + JS 写队列，不再另开 exclusive 连接
- `getRrankerDatabase` 打开失败时重置单例，避免永久卡在失败 Promise
- 单测断言不再出现 `journal_mode`，并适配 `withTransactionAsync`

## 期望输出

- 总览冷启动并发读成绩与曲库时不再因 WAL/双连接互锁卡住
- 好友码同步轮询可正常推进；相关单元测试通过

## 实际输出

- `vitest run`：76 个文件、386 项测试全部通过
