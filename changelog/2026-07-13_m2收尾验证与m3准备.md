# M2 收尾验证与 M3 准备

## 改动原因

用户确认 M2 双端验证交付 OK，并要求再做一次收尾验证后准备进入 M3。需要以当前提交和全新依赖安装重新确认交付基线，同时固定 M3 的范围、批次、隐私与验证门禁，避免未经确认就开始实现。

## 具体实现

- 从 `package-lock.json` 全新执行 `npm ci`，随后复跑 Vitest、Jest、TypeScript 类型检查和 ESLint。
- 重新执行 Android Hermes 静态导出并清理临时导出目录。
- 核对 M3 相关代码与依赖，确认当前没有收藏/练习清单/备份/分享/推荐实现，也没有直接安装文件分享或视图截图依赖。
- 新增 `docs/m3-readiness.md`，按 M3A 本地收藏与备份、M3B 安全导出、M3C 可解释推荐固定开工门禁。
- 更新 `docs/development-readiness.md`、本地 `ROADMAP.md` 与 `TODO.md`，正式标记 M2 完成并记录 M3 的目标、用户价值、实现思路、依赖/风险、优先级和暂缓原因。
- 执行 `npm audit` 核对依赖风险，不自动运行会跨越 Expo SDK 大版本的强制修复。
