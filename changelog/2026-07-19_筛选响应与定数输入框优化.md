# 筛选响应与定数输入框优化

## 改动原因

曲库与成绩页一次持有的歌曲、成绩数据较多，点击筛选标签后同步筛选、排序和列表更新会占用 JavaScript 主线程，使标签选中反馈出现延迟。安卓端定数输入框还受到固定高度和字体内边距影响，“下限”“上限”占位文字存在垂直错位和裁切。

## 具体实现

- 使用 React 延迟值将筛选控件的即时状态与列表筛选计算解耦，曲库页在新结果计算期间显示“正在筛选…”反馈。
- 将曲库列表、歌曲行、歌曲谱面标签、成绩列表和成绩卡片记忆化，并稳定用户曲库操作回调，避免无关状态变化导致可见卡片整批重绘。
- 调整曲库与成绩 FlatList 的首批数量、批次大小、批次间隔和渲染窗口，降低一次提交大量卡片造成的卡顿。
- 安卓定数输入框改用 44 像素最小高度，明确字号、行高、垂直居中和字体内边距设置，去除容易裁切文字的固定高度。
- 曲库与成绩列表将结果和来源元数据封装为同一份渲染数据，避免异步数据切换或热更新期间读取已变为空值的来源对象。
- 增加输入框样式回归断言，覆盖安卓文字完整显示所需的关键属性。

## 期望输出

- 点击难度、版本和类型标签时选中态立即反馈，较重的筛选和列表更新不阻塞本次交互。
- 大数据量筛选后列表分批渲染，减少首帧工作量，滚动和收藏操作保持正常。
- 安卓端“下限”“上限”占位文字完整且垂直居中。
- 异步数据加载和切换期间曲库、成绩页不因来源信息为空而崩溃。
