# M2 首轮双端反馈修复

## 改动原因

用户在 iPhone Expo Go 与 Android Emulator 首轮验收中发现：数字输入框强制使用系统数字键盘；Rating 缺少定数与理论达成率边界提示；牌子页双端在加载期读取未定义数据而崩溃；版本工具把“国服/日服名称对照”误做成了双版本成绩比较。

进一步复验 LXNS 公共端点后确认，姓名框真实响应使用 `required`，且歌曲是包含 `id/title/type` 的对象，并非原实现假定的 `requirements + number[]`。

## 具体实现

- 移除所有数值输入框的 `decimal-pad`，恢复用户常用输入法；Rating 数值解析支持 NFKC 全角数字和逗号小数点。
- Rating 定数限制为大于 0 且不超过 15；达成率仅在超出理论最高 101% 时判错；100.5% 到 101% 仍是有效达成率，并明确提示 Rating 按 100.5% 封顶。
- 牌子页把未加载状态统一转换为空数组，不再直接读取 `data.plates`；渲染只消费 QueryStateView 已传入的稳定快照。
- LXNS 姓名框契约改为解析 `required[].songs[].id/type`，进度同时匹配歌曲 ID、SD/DX、难度、rate、FC 和 FS；缓存 schema 升至 v2，使旧错误缓存安全失效。
- 牌子页只展示带成绩要求的姓名框，不再把纯装饰姓名框当作进度项目。
- 版本页新增 20 个已核实版本的国服/日服名称表；原有统计改成按版本从新到旧排列的单列游玩总结卡片。

## 期望输出

- 数值输入不再强制切换系统数字键盘。
- 定数 15.1、达成率 101.1% 显示明确错误；达成率 100.6% 正常计算并提示 Rating 封顶。
- 牌子页在初始加载、在线响应、缓存响应下均不崩溃，并按真实谱面类型计算。
- 版本页清楚区分“名称对照”和“游玩总结”，其中 `舞萌DX 2026` 对应 `maimai でらっくす PRiSM PLUS`。

## 实际输出

- 通过真实 LXNS `/plate/list?required=true` 验证顶层 `plates`、条件字段 `required` 与歌曲对象结构。
- 通过 LXNS `/song/list` 与水鱼 `/music_data` 的发布歌曲交叉统计核实 20 组版本名称。
- `npm test`：Vitest 19 个文件、64 项通过；Jest Expo 3 个套件、11 项通过。
- `npm run typecheck`：通过。
- `npm run lint`：通过，无警告。
- Android Hermes 静态导出成功，1559 个模块完成打包；验证后已清理导出目录。
- 仍需用户在 iPhone Expo Go 与 Android Emulator 重新打开牌子、Rating 和版本页面完成真机复验。
